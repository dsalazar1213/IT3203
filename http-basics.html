Request and Response Model

HTTP uses a request and response model. A client, usually a browser, sends a request for a resource. The server processes that request and returns a response with content or an error message. In my research paper, I described HTTP as an application layer protocol that makes structured communication possible between browsers and servers. That pattern is repeated constantly while users move through websites.

Simple example: A browser asks for a page, image, or script. The server answers with the requested resource and a status code that tells the browser what happened.

Common HTTP Methods

  • GET retrieves data from the server.
  • POST submits data such as a form or login information.
  • PUT updates or replaces a resource.
  • PATCH updates part of an existing resource.
  • DELETE removes a resource.

My Assignment 1 paper also explains that method permissions matter in security. If a server does not correctly enforce method rules, attackers may try to update or remove resources without authorization.

Important Status Codes

Status Code Meaning Why It Matters
200 Success Shows that the request worked correctly
301 Moved permanently Used for redirects
401 Unauthorized Can point to failed authentication
403 Forbidden Shows access was denied
404 Not found Shows a missing page or file
500 Server error Shows that something broke on the server

Status codes are useful in both troubleshooting and security monitoring. My paper notes that repeated 401, 403, or 404 responses may signal suspicious activity such as brute force attempts or reconnaissance.

Statelessness and Security

One of the most important features of HTTP is that it is stateless. That means each request stands on its own, and the server does not automatically remember previous requests from the same user. Developers use cookies, tokens, and session identifiers to keep track of users across multiple requests.

Statelessness is useful because it helps the web stay simple and scalable, but it also creates security concerns. If an attacker steals a session identifier, that person may be able to impersonate a user. This is one reason why HTTPS, secure cookies, and proper session controls are so important.
